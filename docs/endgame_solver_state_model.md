# Endgame Solver State Model (PIMC + Alpha-Mu)

This document describes what information enters the Rust endgame solvers and what is currently *not* modeled.

Scope:
- `pimc_choose_card(...)`
- `alpha_mu_choose_card(...)`
- `double_dummy::DDState`

## 1) Inputs Taken Directly From GameState

At solver entry (`pimc` / `alpha_mu`), the following are read from `GameState`:

- `hands[viewer]` (viewer's exact hand)
- `hands[p]` sizes for all players (card-count constraints)
- `tricks` (completed trick list, with `(player, card)` per trick)
- `current_trick` (partial trick cards + lead player)
- `current_player`
- `tricks_played()`
- `roles` (declarer/partner/opponent)
- `contract`
- `played_cards`
- `put_down`
- `talon`

These are used to:
- compute legal moves for the viewer
- determine known cards vs unknown cards
- build sampled worlds
- build DD states for exact endgame search

## 2) Generated / Inferred Features

### 2.1 Per-player suit void matrix

Generated by `detect_voids(gs)`:
- `voids[player][suit] = true` when player failed to follow a led suit
- uses both completed `tricks` and `current_trick`

Important details:
- if lead card is Tarok and a follower does not play Tarok, that player is marked void in Tarok
- a player playing Tarok on a suit lead is marked void in that suit

Implementation note:
- void channels are now 5-wide: 4 suits + 1 Tarok channel

This is the main hidden-information inference currently used.

### 2.2 Consistent world samples

Generated by `sample_world(...)`:
- unknown cards are redistributed among non-viewer players
- hard constraints:
  - each player's hand size must match observed size
  - sampled suit cards must respect `voids[player][suit]`
  - sampled taroks must respect Tarok-void (`voids[player][tarok]`)
- fallback exists: if rejection sampling fails repeatedly, deal ignores void constraints

### 2.3 DD compact state

For each sampled world, `DDState::new(...)` contains:
- full 4 hands for that sampled world
- current trick (possibly partial)
- lead player / current player
- tricks played offset
- roles and contract
- declarer raw points accumulated during DD rollout

DD then applies legal-move rules and exact trick resolution to terminal.

## 3) What the Solver Does NOT Carry Explicitly

Inside DD, full historical trick-by-trick logs are not persisted as a replay buffer.

DD keeps only compact forward-search state:
- remaining hands
- current trick
- lead/current player
- counters and accumulated points

So history influences DD *indirectly* via sampled hands and current trick context, not as full replay features.

## 4) Current Objective / Value Signal

DD and the world-evaluation score use:
- declarer-team raw card points (maximize for declarer team, minimize for opponents)

Not explicitly modeled in DD objective:
- explicit contract bonus/malus utility
- announcement utility shaping
- kontra utility shaping

(Those are handled in full-game scoring, not in DD search value.)

## 5) Alpha-Mu Specific Notes

Alpha-Mu currently adds:
- strategy-fusion behavior at root via per-world outcome vectors (same root move across worlds)
- Pareto-front machinery for combining world outcomes

Current implementation detail:
- depth parameter is exposed, but recursive Min/Max front expansion is not fully implemented yet
- practical strength currently comes mainly from root-level world-consistent move choice + DD leaves

## 6) High-Value GameState Descriptions Not Modeled Yet

Candidate additions that may improve endgame inference quality:

1. Trick-winner history features:
- who took each completed trick
- seat-relative control tempo (how often each side retained lead)

2. Tarok exhaustion / dominance descriptors:
- per-player known/possible high-tarok ranges
- remaining trump-overtrump capability estimates

3. Stronger world constraints beyond voids:
- winner-based constraints (cards that must/must-not be in a seat due to historical outcomes)
- partner-call consistency constraints when king call is unresolved

4. Contract-aware DD utility:
- blend raw points with contract utility proxy during search
- optional mode-dependent utility (team contracts vs special modes)

5. Uncertainty quality metrics:
- world diversity / entropy score
- confidence flag when sampled worlds collapse to near-deterministic assignments

6. Explicit lead-control features:
- expected leader for next k plies across worlds
- side-to-control probability in each branch

## 7) Practical Summary

Today, the solver models:
- exact legal move rules
- exact card-level endgame resolution in sampled perfect-information worlds
- per-player suit voids from observed play
- card-count consistency

Today, it does not yet model rich historical semantics such as:
- explicit trick-taker pattern reasoning
- deeper winner-derived hidden-card constraints
- contract-utility-aware DD objective

That makes the current system strong in late-game card-counting and tactical resolution, with room to improve strategic hidden-state interpretation from history.
